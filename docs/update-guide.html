How it works

Every time the app starts it silently contacts the GitHub Releases API in the background. If a newer version exists, the user sees a popup. The entire pipeline — building, signing, publishing, and delivering — is automated. You only need to run two commands.

You push
git tag v1.1
→
GitHub Actions
builds all 3 platforms
→
Binaries are
signed & published
→
User's app
detects update
→
User clicks
Update & restarts
Signature verification Every binary is signed with your private minisign key. The app verifies the signature before applying any update — a tampered or fake binary is rejected automatically.
Part 1 of 3

One-time setup

Do this once. You will never need to repeat it.

1

Install minisign

Minisign is the command-line tool that signs your release binaries. Choose your operating system:

Windows

winget install jedisct1.minisign

Alternatively, download the .zip from github.com/jedisct1/minisign/releases, extract it, and add the folder to your system PATH.

macOS

brew install minisign

Linux

sudo apt install minisign

Confirm the installation worked:

minisign -v

You should see a version number printed. If you get "command not found", restart your terminal and try again.

2

Generate your signing key pair

Run this command once in any folder (your Desktop is fine):

minisign -G

It will ask you to set a password. Choose a strong one — you will need it whenever you sign a release. Save it in a safe place.

This produces two files:

File Purpose Keep secret?
minisign.pub Public key — goes into the app source code No — safe to share
minisign.key Private key — used by GitHub Actions to sign binaries Yes — never commit this

Open minisign.pub in a text editor. It looks like this:

untrusted comment: minisign public key XXXXXXXXXXXXXXXX
RWS5Abc123...long string of characters...

Copy the second line — the one that starts with RWS. That is your public key string and you will need it in the next step.

Never commit minisign.key to git If your private key is exposed, anyone can sign fake updates. Store it only in GitHub Secrets (Step 4).
3

Put the public key into the app code

Open this file in your editor:

internal/updater/updater.go

Find line 22. It currently reads:

const releasePublicKey = ""

Replace it with your public key string from Step 2:

const releasePublicKey = "RWS5Abc123...paste your full RWS... key here..."

Save the file. Every copy of the app now knows how to verify your releases.

Why this matters When a user clicks "Update", the app downloads the new binary and then checks its signature against this public key. If the signature does not match — because the file was tampered with or came from a different source — the update is rejected and the app keeps running the old version safely.
4

Add secrets to GitHub

GitHub Actions needs your private key and password to sign the binaries during the build. Navigate to your repository and follow these steps:

  1. Click Settings in the top navigation bar of your repository.
  2. In the left sidebar, click Secrets and variables → Actions.
  3. Click New repository secret and add the first secret:
Secret name Value
MINISIGN_KEY Open minisign.key in a text editor, select everything, and paste it here.
MINISIGN_PASSWORD The password you chose when generating the key pair in Step 2.
Already wired up The CI workflow file already reads these two secret names. You do not need to change any workflow YAML — just add the secrets with exactly these names.
5

Commit the public key change

Push the code change from Step 3 to GitHub:

git add internal/updater/updater.go
git commit -m "Set release signing public key"
git push

One-time setup is now complete.

Part 2 of 3

Pushing an update

Do this every time you want to ship a new version. Takes about two minutes of your time, then GitHub handles the rest.

1

Bump the version number

Open this file:

cmd/password-manager/main.go

Find line 20:

var Version = "1.0"

Change it to your new version. Use the format MAJOR.MINOR.PATCH — for example:

var Version = "1.1.0"

Use this guide for choosing the right number:

Change typeWhich number to incrementExample
Bug fixPATCH (last number)1.0.0 → 1.0.1
New feature, backwards compatibleMINOR (middle number)1.0.1 → 1.1.0
Breaking change or major rewriteMAJOR (first number)1.1.0 → 2.0.0

Commit and push:

git add cmd/password-manager/main.go
git commit -m "Bump version to 1.1.0"
git push
2

Tag and push the release

This is the single action that triggers the entire automated pipeline — builds, signing, and publishing:

# Create the tag — the version must match what you put in main.go, prefixed with v
git tag v1.1.0

# Push the tag to GitHub — this starts the CI pipeline immediately
git push origin v1.1.0

After pushing the tag, GitHub Actions will:

  1. Run all tests (the build will fail if any test fails)
  2. Build the app binary for Windows, macOS, and Linux
  3. Sign each binary with your minisign private key
  4. Generate a software bill of materials (SBOM) for each binary
  5. Create a GitHub Release and attach all files to it

Watch the pipeline at:

https://github.com/KagisoSetwaba/password-manager/actions

The pipeline usually takes 5–10 minutes to complete.

Tag must match version The tag you push (v1.1.0) must match the version string in main.go (1.1.0). If they differ, the update check will malfunction because the app uses the build-time version to compare against the GitHub release tag.
3

Verify the release on GitHub

Once the pipeline finishes:

  1. Go to your repository on GitHub.
  2. Click Releases in the right sidebar.
  3. You should see v1.1.0 with these files attached:
FileWhat it is
password-manager.exeWindows binary
password-manager.exe.minisigWindows signature file
password-manager.exe.sbom.jsonWindows dependency list
password-manager-macosmacOS binary
password-manager-macos.minisigmacOS signature file
password-manager-macos.sbom.jsonmacOS dependency list
password-manager-linuxLinux binary
password-manager-linux.minisigLinux signature file
password-manager-linux.sbom.jsonLinux dependency list

If all nine files are present, the release is live and users will receive the update notification on their next launch.

Part 3 of 3

What the user sees

No action is required from users beyond clicking one button.

The next time any user opens the app, it silently queries the GitHub Releases API in the background. Within a few seconds of startup, if a newer release exists, a dialog appears:

Update Available

Version 1.1.0 is available.
You are running 1.0.0.

The update will be downloaded, verified, and applied. The app will restart automatically.

Update Now
Later

When the user clicks Update Now, the app:

  1. Downloads the new binary for their operating system.
  2. Downloads the accompanying .minisig signature file.
  3. Verifies the signature against the public key baked into the app. If it fails, the update is aborted and the current version continues running safely.
  4. Replaces the running executable on disk.
  5. Closes — the user re-opens the app and is on the new version.

If they click Later, nothing happens. The check runs again on the next launch.

Quick-reference checklist

Print or bookmark this section. Every release is exactly these four actions.

  • Update the version in cmd/password-manager/main.go
  • git add and git commit the version change
  • git push
  • git tag vX.Y.Z — version must match main.go
  • git push origin vX.Y.Z
  • Wait ~10 minutes and check Releases on GitHub
  • Confirm all nine files are attached to the release

Troubleshooting

Problem Likely cause Fix
Pipeline fails at "Sign binary" MINISIGN_KEY or MINISIGN_PASSWORD secret is missing or wrong Re-add both secrets in GitHub Settings → Secrets → Actions
Pipeline fails at "Test" A test is broken in the new code Fix the failing test before pushing the tag
Users do not see the update popup Tag name does not match version in main.go, or release has no attached assets Check the Releases page; re-tag if necessary
"Signature verification failed" on update releasePublicKey in the app does not match the MINISIGN_KEY secret Ensure both come from the same minisign -G run
Update applies but app crashes on relaunch The new binary was built for the wrong platform, or the build itself had an error Check the Actions log; re-download the correct binary manually from the release page
Accidentally pushed the wrong version tag — Delete the tag locally and remotely: git tag -d vX.Y.Z then git push origin --delete vX.Y.Z, then re-tag correctly
